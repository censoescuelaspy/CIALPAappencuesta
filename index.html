Metodologia de funcionamiento y datos

Como opera CIALPA de punta a punta: herramientas usadas, interconexion, captura offline, resguardo formal, integridad y trazabilidad.

Modelo operativo CIALPA

La app combina captura de campo, respaldo local, backend auditable y una ruta de base de datos transaccional.

El flujo actual conserva la operacion simple con Google Sheets y Drive, pero ya deja preparada la maduracion hacia PostgreSQL mediante una cola idempotente y una API de sincronizacion. El objetivo es que cada respuesta tenga autor, momento, dispositivo, version del formulario, escuela, institucion, estado y evidencia relacionada.

1dato capturado una vez, reutilizable para campo, gabinete y MEC.
3niveles de resguardo: dispositivo, Sheets/Drive y base formal.
100%de escrituras criticas con usuario, fecha, version y estado.
1

Captura en campo

Encuestador, supervisor o director registra escuela, servicios, bloques, pisos, aulas, sanitarios, exteriores, fotos, GPS y tiempos.

Tablet, celular, notebook

Backend auditable

5

Repositorio operativo

Sheets conserva datos tabulares y Drive organiza fotos, anexos, JSON y paquetes de sincronizacion.

Sheets, Drive, LOG

Cola formal

Base transaccional

La ruta de maduracion normaliza escuelas, instituciones, bloques, ambientes, sanitarios, evidencias y auditoria en PostgreSQL.

Cloud SQL, Supabase o AlloyDB

Ciclo metodologico del dato

1. IdentificacionSe selecciona escuela/local escolar desde padron, asignacion o enlace publico. Se fijan codigo local, nombre, departamento, distrito, localidad, usuario y version de app.
2. Captura guiadaEl Registro guiado pregunta por etapas: ubicacion base, perimetro, bloques, pisos, aulas, sanitarios, exteriores, servicios, electricidad, evidencias y revision final.
3. AutoguardadoCada avance actualiza el borrador local. Si no hay conexion, el registro queda en cola local y se sincroniza al volver Internet.
4. SincronizacionLa capa API envia el paquete al Web App de Apps Script. El backend valida token, permisos, columnas y estado antes de escribir.
5. Resguardo operativoSheets recibe filas consultables; Drive recibe fotos y JSON; LOG/ERRORES registran eventos, fallas y trazabilidad tecnica.
6. Consolidacion formalEl paquete puede reprocesarse desde db_sync_queue, mec_borradores o JSONL con npm run db:consolidate.
7. Uso institucionalMapa, tableros, planificacion, auditoria, reportes y exportaciones consultan datos con filtros por territorio, escuela, institucion, fecha, usuario y estado.

Herramientas y responsabilidades

  • GitHub Pages: publica la app estatica y permite cache-busting por version.
  • Service Worker: conserva app shell, evita pantalla en blanco y renueva cache con Actualizar app.
  • JavaScript PWA: coordina login, mapa, formularios, plano, tableros, cola offline y sincronizacion.
  • Google Apps Script: funciona como API auditable y puente de integracion con Sheets, Drive y la base formal.
  • Google Sheets: base operativa visible para administracion, control rapido y contingencia.
  • Google Drive: resguarda fotos, anexos, PDFs, JSON completos y payloads de sincronizacion.
  • PostgreSQL: destino formal para integridad referencial, auditoria, consultas y escalamiento.
  • Consolidador DB: reenvia exportes historicos y errores de cola a la API transaccional con idempotencia.
  • Leaflet, OSM y Canvas: mapa territorial, base mapa de escuela y plano tecnico editable.

Integridad, consistencia y trazabilidad

  • Cada registro debe tener id_registro, app_version, usuario, rol, fecha cliente y fecha servidor.
  • Cada sincronizacion debe usar clientMutationId para que un reintento no duplique filas.
  • El backend valida permisos por rol y escuela asignada; la interfaz solo ayuda, no es la autoridad.
  • Las entidades se relacionan por school_key, institution_key, bloque, piso, ambiente, sanitario y evidencia.
  • Los cambios sensibles quedan en LOG/ERRORES y, al migrar, en audit_log inmutable.
  • Los cierres finales no deben degradarse a borrador por autoguardados posteriores.
  • Las fotos quedan vinculadas a escuela y entidad tecnica; no se guardan como archivos sueltos sin contexto.

Estados del dato

  • BORRADOR LOCAL: existe solo en el dispositivo y puede editarse.
  • PENDIENTE_SYNC: ficha completa o parcial esta en cola por falta de conexion.
  • RECIBIDO: Apps Script confirmo escritura en Sheets/Drive.
  • EN_COLA_DB: el paquete quedo registrado para base formal.
  • SINCRONIZADO_DB: PostgreSQL aplico transaccion e idempotencia.
  • VALIDADO / OBSERVADO / ANULADO: supervision institucional decide consistencia tecnica y vigencia.

Controles tecnicos

  • Validaciones frontend para campos obligatorios, formatos, coordenadas, montos, fechas y dependencias.
  • Validaciones backend para token, rol, escuela, encabezados, duplicados y estado permitido.
  • Reglas condicionales para no enviar campos ocultos cuando la respuesta base no corresponde.
  • Cache de catalogos y padron para evitar bloqueos si el backend no responde.
  • Revision por tableros: inconsistencias, alertas territoriales, evidencias, tiempos y cobertura.
  • Versionado de app y Service Worker para reproducir con que codigo fue levantado cada dato.

Ruta de maduracion hacia base formal

Fase 1 - Operacion actual GitHub Pages + Apps Script + Sheets/Drive + offline local, con LOG y evidencias.
Fase 2 - Doble escritura Guardar en Sheets y en db_sync_queue; reenviar a API PostgreSQL sin bloquear campo.
Fase 2B - Consolidacion historica Usar npm run db:consolidate para pasar exportes de cola, borradores o JSONL a PostgreSQL y comparar conteos.
Fase 3 - Fuente oficial PostgreSQL pasa a fuente primaria; Sheets queda como exportacion y control operativo.
Fase 4 - Gobierno nacional Backups, retencion, vistas analiticas, catalogo de datos, auditoria completa y reportes MEC.
